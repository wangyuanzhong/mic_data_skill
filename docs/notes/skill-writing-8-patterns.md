# Skill 写作思路笔记：8 种常用手段

> 给自己看的笔记。核心就一句：  
> **用户经常说不清；Skill 要把 Agent 容易乱跑的地方写死。**  
> 下面 8 种不是「格式规定必须有」，而是写正文时常用的招。可以混用。

---

## 先分清：格式必须有 vs 写作手段

| 类型 | 有什么 |
|------|--------|
| Skill **格式**强制要求 | 文件夹里有 `SKILL.md`；开头有 `name`、`description` |
| 下面 8 种 | 正文里怎么写才稳，按需要选用 |

`description` 像门铃（什么时候用）；正文像 SOP（怎么干）。

---

## 1. 表格 + 硬规则（收信息、卡住别乱来）

**直白说：**  
把「要问清什么」列成表；把「什么情况下必须停」写成不许破例的规矩。

**适合：** 用户说法乱七八糟，但入口信息其实就那几项。

**例子（我们的频响 Skill）：**

- 表格：路径必填；产品名可选……  
- 硬规则：没路径只问路径，禁止开始瞎分析；不许让用户重写一整段标准提示词。

**反例：** 只写「请充分理解用户需求」——等于没写，Agent 还是会猜。

---

## 2. 填空模板 / 输出合同（规定成品长什么样）

**直白说：**  
别让 Agent 每次写成不一样的作文；章节、文件名、必出图写死，它只管填空。

**适合：** 怕报告/结论结构乱、今天一个样明天一个样。

**例子：**

- 报告必须有：`## 1. 运行追溯` … `## 5. 曲线标准`，不能删章、不能调序。  
- 图必须叫 `fig_curve_golden_overlay.png`，缺图 = 没做完。

**生活比喻：** 医院病历首页格式是固定的，医生填内容，不能自己发明一种病历。

---

## 3. 检查清单 Checklist（一步步勾）

**直白说：**  
用 `- [ ]` 列出步骤，做完一项勾一项，减少漏步。

**适合：** 流程长，容易跳步。

**例子：**

```text
门禁1：
- [ ] 表A 文件清单已展示
- [ ] 表B 数据预览已展示
- [ ] 表C 映射已让用户确认
- [ ] 用户确认前不写结果、不算金标
```

**和硬规则的差别：** 清单偏「提醒做全」；硬规则偏「违反了就不许继续」。

---

## 4. 好坏对照 Examples（给对的和错的）

**直白说：**  
Agent 有时听懂道理但不会落地；直接甩一个对、一个错，它模仿对的。

**适合：** 沟通方式、提问方式、文案口吻这类「感觉题」。

**例子（门禁1）：**

- ❌ 错误：「请问哪一列是频率？」（用户可能也不知道）  
- ✅ 正确：「我猜第1列是频率，预览是 20、25、31.5… 对吗？不对请直接改。」

**再举一个提交信息：**

- ❌ `update`  
- ✅ `fix(report): 修正 1kHz 灵敏度单位显示`

---

## 5. 决策表 / 短流程（若 A 则 B）

**直白说：**  
写清分支：遇到这种情况走左边，那种走右边；或固定顺序不许颠倒。

**适合：** 步骤顺序错了结论就错。

**例子（奇异值）：**

```text
先算均值 → 再判断奇异值 → 用户确认剔除 → 再重算均值
（禁止：没算均值就先乱删曲线）
```

**再例（多产品）：**

```text
若文件夹里多个产品 → 先出拆分表让用户确认 → 再按产品一个一个跑
（禁止：混在一起算一条平均曲线）
```

---

## 6. 禁止清单 + Red Flags（专治找借口偷懒）

**直白说：**  
提前写出 Agent 会怎么给自己开脱；一冒出这些念头就停、重来。

**适合：** 规矩 Agent「其实懂」，但一赶时间就跳过。

**例子：**

```text
Red Flags（出现就要停）：
- 「太简单了，不用确认映射」
- 「先算完金标再问用户」
- 「用户催得紧，跳过门禁」
- 「高Q尖峰看起来怪，先当坏样删了」← 我们规定 Q≥10 不算奇异值
```

**和硬规则的差别：** 硬规则写「必须怎样」；Red Flags 写「脑子里这句话就是危险信号」。

---

## 7. 渐进披露（正文短，细节外挂）

**直白说：**  
`SKILL.md` 只留主干；又长又细的说明放到旁边的 `references/`，用到再读。

**适合：** 领域知识很多，全塞进正文会又臭又长，还占上下文。

**例子：**

- `SKILL.md`：步骤1要识别布局，已知三种代号是 A/B/C。  
- `references/layout-catalog.md`：`456_FR.xlsx` 第几行是 Curves、`.SPL` 文件头怎么跳过……

**生活比喻：** 总手册一页流程图；厚附录放抽屉里，需要再翻。

---

## 8. 脚本当计算器 + 默认值策略

分两小点，常一起出现。

### 8a. 脚本（算的事别让模型临场发明）

**直白说：**  
排名、平均、分档这种「每次必须算成一样」的，交给固定程序；Skill 写「何时调用、结果当什么用」。

**例子：**

- Skill：确认映射和剔异名单后，运行分析脚本。  
- 脚本：算平均、δ 包络排名、0.5dB 灵敏度分档，写出 `process.xlsx`。  
- Agent：读表写报告，**不许手改数字**。

### 8b. 默认值（用户没说时用什么）

**直白说：**  
把「没说清时的默认」写进 Skill，减少反问次数，行为仍稳定。

**例子：**

- 用户没说目标 → 默认做完整条流水线。  
- 用户没说辅标要几个 → 默认取曲线排名第 2、3 名。  
- 用户没说频段 → 默认 20Hz–20kHz（门禁2仍可改）。

## 怎么选？（速查）

| 你怕什么 | 优先用 |
|----------|--------|
| 用户说不清、信息收不齐 | 1 表格+硬规则；8b 默认值 |
| 报告每次换皮 | 2 模板 |
| 流程漏步 | 3 清单；5 短流程 |
| 提问/口吻不对 | 4 好坏对照 |
| 明知故犯跳门禁 | 6 Red Flags |
| 正文太长 | 7 外链参考 |
| 数字每次算不一样 | 8a 脚本 |

## 和我们这个频响 Skill 的对应（备忘）

| 手段 | 用在哪 |
|------|--------|
| 1 表格+硬规则 | 步骤0 Intake；各门禁「未确认不许继续」 |
| 2 模板 | `report.md` 固定章节 + 规定图文件名 |
| 3 清单 | 门禁1 表 A–D |
| 4 好坏对照 | 门禁1 怎么问用户 |
| 5 短流程 | 先均值→剔异→重算；多产品先拆分 |
| 6 Red Flags | 禁止跳过映射/剔异确认 |
| 7 外链 | 布局字段说明 → `layout-catalog.md` |
| 8 脚本+默认 | δ 排名、0.5dB 分档；默认整条流水线 |

---

## 一句话收尾

Skill 不是散文，是给 Agent 的约束。  
**用户说不清不可怕；可怕的是 Skill 也写不清。**  
上面 8 招都是在把「说不清」变成「可检查」。
